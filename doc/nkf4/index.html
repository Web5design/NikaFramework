This document is official documentation for Nika Framework 3.

Intro

"Nika Framework" is Architectural Front-End JavaScript Framework. Unlike others, it archives different goals. It's not UI framework, as it's doesn't provide any UI components and widgets. It's also doesn't MVC one, as it has different architecture. This framework allows you to use any UI components as you wish, and use any JavaScript libraries. All that it provide you - it's powerful architectural style.

So what is NKF? It's widget based framework, that allows you to build complex application based on widgets.

Terms

UI

User Interface

NKF

Nika Framework

Abstract Component

Some type of the following one: layout, page, widget, component

Layout

Page layout correspond for components placement

Page

Component that has predefined components placement

Widget

Some UI part on page that corresponds some logical entity

Features

  • Full AJAX. All page changing - only via AJAX
  • One request to server per page. In this request you get all needed data to your components
  • Event Driven Development (EDD). All Components are absolutely independed and communicate each other via events.
  • Components reusable. You can use a few times on some page the some component with different view/functionality/etc...
  • Two files on output: merged.js and merged.css. For production mode merged.jsz and merged.cssz.
  • XTML, SASS, SVG support.

Structure - Details Overview

Lets considering detailed, why we need these folders and how we can use files from them.

dom folder used to store your own xhtml piece of code. Later you can access them via inherited method _getComponent. 

    this._getComponent();

Using method without any parameters returns dom/index.xhtml by default.

Note

Please note, that by default component type is xhtml files. How to access another types - please read below.

To access files in subfolder you need to pass componentPartName key. 

    this._getComponent({
     componentPartName: "items/controlItem"
    });

will returnsdom/items/controlItem.xhtml. If you what to access for example you svg files, you need to pass componentPart key. 

    this._getComponent({
     componentPart: "svg",
     componentPartName: "items/controlItem"
    });

in this case it returns yousvg/items/controlItem.svg.

Note

Please note, that when you access your components with _getComponent method, it returns it jQuery-wrapped. It means, that you don't need to wrap it again, you can use all jQuery methods on it right after accessing.

Build System

To understand how later all files from component build, lets considering how build system works.

Build System relay on impl/conf/include.list file, where you point files that should be included. In most cases it's only JavaScript files, and could be one sass file, that include others. 


    impl/extra/style/main.scss

    impl/extra/js/jquery.js
    impl/extra/js/plugins/jquery.attributes.js
    impl/extra/js/plugins/jquery.namespace.js
    impl/extra/js/plugins/string.format.js
    impl/extra/js/plugins/jquery.cookie.js

    impl/extra/js/base64.js

    common/network/NetworkAbstract.js
    common/network/NetworkOnline.js
    common/network/NetworkManager.js

    common/storage/StorageAbstract.js
    common/storage/MemoryStorage.js
    common/storage/CookieStorage.js
    common/storage/LocalStorage.js
    common/storage/StorageManager.js

    .....

This file need due to right order JavaScript files includes. As for example, a.js could relay on b.js, so b.js should be included first.

  • So, first Build System runs over include.list and include all files that mentioned there.

  • Next, it runs over impl folder and pick up all.xhtml,.svg, .css and .css files

  • Than, all scss files convert to css files.

  • All images pointed in .xhtml and .css files converted to data:base64 protocol.

  • All .xhtml files merged into hashmap and available via global variable __dom__

    Note

    You don't have to access that variable directly. Use _getComponent method instead.

  • And in the end - generates two files: merged.css and merged.js

Build System works in two modes: development andproduction. By default it runs in development mode, but if you pass second parameter as production it will create compressed versions of .js and .css, like .jsz and .cssz.

Build System located in build folder and written in NodeJS. 

                $ node build.js # runs in development mode
            
                $ node build.js production # runs in production mode

Component Structure

Lets start with JavaScript structure. 

    (function() {
      "use strict";

      var ns = "ks.nkf3.impl.components.layout";
      var self = $.namespace(ns);

      extendClass(Login, ks.nkf3.common.components.component.LayoutAbstract);
      Login.className = "Login";

      function Login() {
        this.className = Login.className;

        this.getRenderedDOM = function() {
          return domComponent;
        };

        function constructor() {
          init.call(this);
        }

        function init() {
        }

        var domComponent = this._getComponent();

        constructor.call(this);
      }

      $.extend(self, {
        Login: Login
      });

    })();

As we can see, each component has to implement appropriate super class. For layout it's LayoutAbstract, for page it's PageAbstract and so on...

Also, we can see that there's a namespace using. In out case it's ks.nkf3.impl.components.layout. Each component must be in appropriate namespace.

className using for prevent class names while obfuscare.

Also, we must to override getRenderedDOM method for each component. This method should return ready DOM for component.

Another it's optional.

First Application

Lets create a sample application, that allows us to understand how NKF works.

Lets start from Layout. We will use login layout. You can call it whatever you want. In our case, it'll responses for showing header and footer.

Create in impl/components/layout Login folder. Inside this folder create three folders: dom, style, login.

In logic folder create file called Login.js with the following content: 

    (function() {
      "use strict";

      var ns = "ks.nkf3.impl.components.layout";
      var self = $.namespace(ns);

      extendClass(Login, ks.nkf3.common.components.component.LayoutAbstract);
      Login.className = "Login";

      function Login() {
        this.className = Login.className;

        this.getRenderedDOM = function() {
          return domComponent;
        };

        function constructor() {
          init.call(this);
        }

        function init() {
        }

        var domComponent = this._getComponent();

        constructor.call(this);
      }

      $.extend(self, {
        Login: Login
      });

    })();

In fact, this file do anything but returns DOM for Login layout.

Now, lets create DOM for Login. For that, create dom/index.xhtml file with the following content: 

    
    <div>
        <header>
            <div data-nkf-component-type="widget" data-nkf-component-name="Header"></div>
        </header>

        <section></section>

        <footer>
            <div data-nkf-component-type="widget" data-nkf-component-name="Footer"></div>
        </footer>
    </div>

Please pay attention, that we use data-nkf-component-* attributes. Due to them, we define what components will be use. data-nkf-component-type - corresponds for component type, e.g. layout, page, widget, component. data-nkf-component-name - component name.

Please note, that we have section tag. It required, as in this tag will be written page.

Optionally you can create a style for component in style folder.

So, when framework will render this layout, it'll always show header and footer even when you will change content. As all that pointed in layout will never change, until you change layout.

In same way, you can create a page. The difference is that it has to be inherited from PageAbstract. In DOM you can also defined some widgets that has to be shown.

And in the same manner, you what to create defined widgets. In most cases, actually widgets it's the most complicated components in system.

Lets considered one: 

    (function() {
      "use strict";

      var ns = "ks.nkf3.impl.components.widget";
      var self = $.namespace(ns);

      extendClass(Header, ks.nkf3.common.components.component.WidgetAbstract);
      Header.className = "Header";

      function Header() {

        // --------------------------------------------------------------------
        // Public methods/variables
        // --------------------------------------------------------------------

        this.className = Header.className;

        this.getRenderedDOM = function() {
          return domComponent;
        };

        // --------------------------------------------------------------------
        // Private methods
        // --------------------------------------------------------------------

        function constructor() {
          init.call(this);
        }

        function init() {
          $("#customer-logo img", domComponent).attr({
            src: "/nkf3/impl/extra/img/customers/{customerName}.png".format({
              customerName: globalData.main.customerNickName
            })
          });
        }

        // --------------------------------------------------------------------
        // Private variables
        // --------------------------------------------------------------------

        var domComponent = this._getComponent();
        var widgetData = this._getData();

        var $ComponentManager = ks.nkf3.common.components.ComponentManager.getInstance();
        var globalData = $ComponentManager.getGlobalComponentsData();

        constructor.call(this);
      }

      $.extend(self, {
        Header: Header
      });

    })();

This is quite simple widget, that insert into predefined DOM customer logo.

Please paint attention to _getComponent and _getData methods. _getComponent - in this case will get default .xhtml file, in our case it's index.xhtml. But _getData method without any parameters will get data for current widget from JSON page file. Now lets move further, and look at $ComponentManager variable. In NKF all system variables that refer to core functionality, writes with dollar sight.

ComponentManager - it's class, that manage components render, page loading and other work that should be manage. In this case, we call getGlobalComponentsData method to fetch global component data.
